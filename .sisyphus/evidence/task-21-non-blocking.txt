it('should not block the main thread during analysis', async () => {

    visionRateLimiter: mockRateLimiter,
    featureFlags: mockFeatureFlags
  });

  await vp.start();

  // Start a loop iteration

## Architecture Analysis: Why VisionProcessor Does Not Block Pilot

### 1. Separate Event Loop
- VisionProcessor uses its own setTimeout chain
- Pilot uses its own setTimeout chain
- Both are on the same Node.js event loop but never await each other
- Vision loop timing: 2-10 seconds (much slower than Pilot's 200-2000ms)

### 2. No Shared State
- VisionProcessor does NOT read or write to state files that Pilot uses
- VisionProcessor stores analysis in its own `latestAnalysis` property
- No file I/O in the vision loop that could block Pilot's file operations

### 3. Async Design
- VisionProcessor.loop() is async and uses await
- The await yields to the event loop, allowing Pilot to run
- Rate limiter uses Bottleneck (separate async scheduling)
- No synchronous blocking operations in the loop

### 4. Minimal Overhead
- getBotState() is purely synchronous (reads bot entity properties)
- captureScreenshot() is synchronous (returns object, no actual rendering)
- Only the schedule() call is async (through Bottleneck rate limiter)
- Total sync work per cycle: <1ms (negligible compared to Pilot's 200ms minimum)

### Conclusion
✅ VisionProcessor is architecturally non-blocking relative to Pilot
